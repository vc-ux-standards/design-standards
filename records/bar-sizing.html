Bar thickness
and spacing

A bar's thickness is not a style knob — it follows the width and the count, within limits. Let the bars scale to the plot, but clamp them: never thinner than the eye can read, never so fat they stop reading as bars. The two failures are the same reflex — letting the chart stretch its bars to fill the container: three categories balloon into slabs, forty collapse into hairlines. Cap the thickness, set the gap as a fraction of the step, and when there are only a few bars, hold the cap and leave the whitespace.

Our context

VC's charts live in dashboard tiles and panels that resize under a grid, and the category count is set by the data, not the designer — a payer mix has three bars, a provider ranking has forty. Left to the charting library's default, bar thickness is whatever fills the width, so the same chart that reads at eight bars becomes three fat slabs when the data returns three, or an unreadable comb when it returns forty. Thickness has to be governed by rule, not by the container.

The rule (provisional)

Bar thickness is variable, within a floor and a ceiling — never a fixed pixel width, never left to fill the container.

  • Scale, then clamp. Thickness = the band (plot width ÷ count) × a fixed fill fraction, clamped to a floor of ~8px — below it a bar can't be read or tapped, so change the form — and a ceiling of ~48px for a single series, above which it reads as a block.
  • Gap is a fraction of the step, not a pixel. The bar takes ~two-thirds of the step, the gap ~one-third (roughly 2:1). The gap never beats the bar, and never closes to nothing.
  • Few bars: hold the ceiling, don't stretch. With 2–3 categories, cap each bar at ~48px and leave the whitespace — a single value is a number, not a bar.
01 / Quick reference

Thickness follows the count

Set the count first, then read the thickness off it. The bar scales with the width up to a ceiling and down to a floor; past either limit the answer is a different form, not a thinner or fatter bar. The numbers below are for a single series of vertical columns — for horizontal bars the same limits govern row height and the vertical gap.

Category count Thickness Spacing
1 value Not a bar — show a number with its delta. See charts at tile scale.
2–3 Hold the ~48px ceiling; do not stretch to fill. Fixed gap; leave whitespace beside the group.
4–12 Variable — the band scales the bars to the plot. Gap ~one-third of the step (2:1 bar to gap).
Past ~12 Variable down to the ~8px floor, then stop. At the floor, go horizontal and scroll — see bars and columns.

Three numbers carry the rule.

  • Floor ~8px. Thinner than this and a bar can't hold a colour, a label, or a pointer — the signal that the count has outgrown the form.
  • Ceiling ~48px for a single series. Fatter than this and length stops reading as a quantity; the bar becomes a panel.
  • Gap ~half the bar. The bar takes about two-thirds of each step and the gap the rest — enough to separate, never enough to dominate.
02 / The pattern in practice

The slab, and the gap

Both bad bars come from the same default: stretch to fill. Let the library size the bars to the container and a short series fattens into slabs while the gaps between them vanish. Cap the thickness and fix the gap as a fraction of the step, and both fix themselves.

Avoid
Membership by line · stretched

Three values stretched into slabs. Each bar is ~88px wide because the chart divided the panel by three and filled it. Length still encodes the value, but the eye reads three blocks, not a comparison — width is carrying weight it shouldn't.

Use
Membership by line · capped

Capped at the ceiling, left-aligned, whitespace to the right. The three bars read as bars, the gap is a fraction of the step, and the empty panel is not an error — a short series does not owe the container a full row. Same values, honest thickness.

Avoid
Audits by month · no gap

Bars set edge to edge read as one shape. With the gap squeezed to nothing, the columns fuse into a toothed block — a histogram's look, not a comparison. The reader can't tell where one category ends and the next begins.

Use
Audits by month · 2:1 gap

The bar takes two-thirds of the step, the gap the rest. A gap about half the bar width separates every column without pulling the eye — each reads as its own value. The ratio holds as the panel resizes, because it's a fraction of the step, not a fixed pixel.

03 / The reasoning

Four limits the bar keeps

A bar encodes one value as one length; its thickness carries no data at all. Every limit below protects the reading — keep the thickness in a band where length is legible and comparable, and let the gap group the bars without competing with them.

1

Thickness carries no meaning

Length is the value; width is not. Because thickness encodes nothing, it must never vary with anything but the fit — bounded above and below, and equal across every bar in the series.

2

The floor is a legibility limit

Under ~8px a bar stops working. It can't hold a colour, seat a label, or take a pointer — so the floor is also the signal to change form: go horizontal and scroll, or drop to a table.

3

The ceiling keeps a bar a bar

Past ~48px, length stops reading as quantity. A very fat bar is seen as an area or a panel, not a measured length — so cap it, and let the plot carry whitespace rather than the bars carry the slack.

4

The gap is a fraction, not a pixel

Spacing set in pixels breaks on resize. Tie the gap to the step — about a third of it — so bars stay dominant and evenly separated at every panel width. For grouped bars, tight within a group, looser between.

04 / What the field says

The knobs already exist

Every serious charting library exposes exactly these controls — a max bar width and a gap set as a fraction — which is the precedent for governing them. The defaults stretch to fill; the settled practice is to cap the width and express the gap as a proportion of the step.

D3 · scaleBand

The gap is a share of the step

D3's band scale sets spacing with paddingInner and paddingOuter — fractions from 0 to 1 of each step given to the gap, not pixel values. It is the canonical model: bars and gaps are proportions of the band, so they hold as the range resizes.

gap as a fraction of the step
Apache ECharts

Cap the width, size the gap by percent

ECharts caps thickness with barMaxWidth and spaces categories with barCategoryGap, whose default is '20%' of the category width; barGap separates series. Width has a ceiling; spacing is a percentage.

barMaxWidth · barCategoryGap
Highcharts

A maximum point width

Highcharts exposes maxPointWidth so a column can't balloon when the category count is small, with pointPadding and groupPadding given as fractions of the category width. The library ships the exact cap this record asks for.

maxPointWidth · fractional padding
Google · Material 3

Consistent spacing for comparison

Material treats the bar chart as a category comparison and calls for even, consistent spacing between bars so the set reads as one series. It stops short of pixel figures, but the direction — regular gaps, comparable bars — is the same.

even, consistent spacing
Synthesis

The pattern for tools like ours

For VC's resizing tiles and data-set counts: scale the bars to the band, clamp to a ~8px floor and a ~48px ceiling, and set the gap at about a third of the step. When the count is small, hold the ceiling and leave whitespace; when it hits the floor, change the form. Our usage should follow it.
