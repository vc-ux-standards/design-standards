Labels, descriptions,
hints and placeholders

Four different tools get treated as interchangeable, and the most common mistake is using the most fragile one — placeholder text — to do the most important job: naming the field. This record sets out what each does and when not to reach for it. Draft: argue with it, don't cite it yet.

Our context

VC forms carry consequential, domain-specific fields — NPIs, value-set IDs, attribution windows, measurement periods — where an unlabelled or under-explained field is a real error risk for the analysts and admins filling them in. Our users work fast on desktop and revisit forms; a label that vanishes the moment they start typing, or essential guidance hidden in grey placeholder text, costs them accuracy. We need a clear division of labour between the four pieces of field text.

The rule (provisional)

Placeholder text is never a substitute for a visible label. Every field gets a persistent, visible label. Add a description above the input when the field needs explaining, and an inline hint below it for a format or constraint. Use placeholder text rarely, and only as a faint example of the input — never to carry the field's name or any essential information, because it fails accessibility checks and disappears the instant the user types.

01 / The four tools

Four jobs, four different pieces of text

Each piece of field text does one job and sits in one place. Get the job right and the field explains itself; reach for the wrong one — especially the placeholder — and the field becomes a guessing game.

ToolIts job · where it sitsWhen not to use it
Visible label Names the field. Always present, above the input, and stays put. Programmatically tied to the input. Never optional. There is no field without one.
Description / help text Explains what's expected or why we ask. Sits between the label and the input, so it's read before typing. Don't use for a simple format rule — that's a hint. Don't pad obvious fields with it.
Inline hint States a format or constraint — "10-digit NPI", "max 200 characters". Sits just below the input. Don't hide essential meaning here, and never put it in the placeholder where it vanishes.
Placeholder At most a faint example of a value, shown inside the empty input. Optional, and skippable. Never the label. Never essential info. Never instructions. It disappears on input and reads as filled-in.
02 / The rule

A placeholder is not a label

This is the rule worth tattooing on the form. A label inside the placeholder looks clean in a mockup and breaks in use: it disappears the moment the user types, so anyone who pauses, gets interrupted, or comes back to check loses the name of the field. It fails accessibility — placeholder text isn't a reliable accessible name, fails WCAG 3.3.2 (Labels or Instructions), and its grey colour usually fails contrast. And a filled-looking field reads as already-answered, so it gets skipped.

04 / Reject in review

The labelling anti-patterns

  • Placeholder as the label. Any field whose name lives only in the placeholder. Add a visible, persistent label.
  • Essential information in a placeholder. Format rules, instructions or constraints that vanish on input. Move them to a hint or description.
  • Label and placeholder saying the same thing. "Email" with placeholder "Email" adds nothing; either drop the placeholder or make it a genuine example.
  • A description crammed into the label. Long explanatory text belongs in help text between the label and the input, not in the label itself.
  • Vague labels. "Name", "Date", "Value" with no qualifier on a domain field. Say which name, which date, which value.
  • Hint hidden until error. A format that's only revealed after the user gets it wrong. Show the constraint up front as a hint.
05 / What the standards say

"Don't use placeholders" is the consensus

The accessibility standard and the major design systems are unusually aligned here: a visible label is required, and placeholder text is explicitly discouraged as a stand-in. Links are in the references.

W3C · WCAG — the standard

Labels or instructions are required

Success Criterion 3.3.2 (Labels or Instructions) requires that labels or instructions are provided when content requires user input. Placeholder text alone does not satisfy it — it isn't a reliable accessible name, and its low-contrast grey commonly fails 1.4.3 as well.

visible label required
GOV.UK Design System

Do not use placeholder text

GOV.UK's guidance is blunt: do not use placeholder text in place of a label or hint. It disappears when users start typing, can be mistaken for a pre-filled value, has poor contrast, and is not consistently announced by assistive technology. Use a label, and a separate hint where needed.

no placeholder labels
Nielsen Norman Group

Placeholders in form fields are harmful

NN/g's testing shows placeholder-as-label increases errors and cognitive load: users forget what the field was, can't check their answer, and skip fields that look filled. The recommendation is a persistent label, with any example shown outside the field.

persistent label
MDN · web standards

Tie the label, describe the rest

MDN documents the mechanics this rule depends on: a <label> associated with its control via for/id, and supplementary description or hint text linked with aria-describedby — not the placeholder — so assistive tech announces it.

label + aria-describedby
Synthesis

The pattern for forms like ours

For consequential, domain-heavy fields, the standard is unambiguous: a visible persistent label on every field, description and hint where they earn their place, and the placeholder demoted to an optional example. VC should follow it.

label always, placeholder rarely
06 / Recommendation for VC

Label everything; placeholder is an example, never a name

The house rule (provisional)

  • Every field has a visible, persistent label tied to its input. No exceptions, including search and single-field forms.
  • Add a description above the input when the field needs explaining or carries consequence; keep it out of the label.
  • Add an inline hint below the input for a format or constraint, shown up front rather than only on error.
  • Use placeholders rarely, and only as a faint, skippable example of a value — never the label, never essential information, never instructions.
  • Reject in review any field whose name or required guidance lives only in placeholder text.
