Descriptions and
placeholders

A field carries two optional slots: a description above the input, and a placeholder inside it. One persists; the other vanishes the instant the user types. Treating them as interchangeable is how essential guidance lands in the one place it can't survive. This record sets out what each carries, and when a field needs neither.

Our context

VC forms carry consequential fields — NPIs, value-set OIDs, attribution windows — where a misread is a real error for the analysts and admins filling them in. The system gives each field exactly two text slots and no third: a description above the input and a placeholder inside it. There is no hint line below. That constraint is the whole decision.

The rule (provisional)

Sort each piece of text by one test — must it survive the user typing?

  • Must survive → the description above the input: what it means, why we ask, its format and constraints. It persists and is announced.
  • Loses nothing → the placeholder: at most a faint example of a well-formed value, never a name, an instruction, or anything essential. It disappears on the first keystroke and isn't reliably read aloud.
  • Self-evident field → neither.
01 / Quick reference

One axis decides it: does the text survive typing?

One axis decides it: is the text still there once the user types? The description is permanent; the placeholder is not — and every other difference follows. Read the table down that axis: what a slot can safely carry is set by whether it disappears.

Placeholder · inside Description · above
Persistence Gone the instant the user types. Returns only if they clear the field. Always visible, before, during and after typing.
When it's read Only while the field is empty; skipped once focused. A standing reference the user can re-check at any point.

02 / The pattern in practice

Put the guidance where it survives

The common failure is loading the placeholder with the work the description should do — the format, the instruction, the thing the analyst actually needs. It looks complete in a static mockup and breaks the moment anyone types. Watch the difference on a field where format matters — a 10-digit NPI. The placeholder's faint grey is the system telling you it's disposable.

Avoid
Add provider

The only statement of the format lives in the placeholder. The instant the analyst types — as in the second field — it's gone, and anyone returning to check the value sees no guidance at all.

Use
Add provider

The 10-digit National Provider Identifier for the billing provider.

The organisation's NPI, if it bills as a group.

Format and meaning sit in the description, which stays put and is announced. The placeholder shows only the shape of a valid value, so nothing is lost when it disappears.

Don't ask the placeholder to do work it can't

A placeholder earns its place rarely, even without instructions. Two patterns to avoid: one that merely repeats the label (clutter, no information); and an example so complete it reads as a value already entered and gets left untouched. When an example genuinely helps, make it unmistakably an example — a truncated sample of the format, not a plausible whole answer.

03 / The reasoning

Four forces, all pointing the same way

The division isn't a style preference. Four pressures act on every field, and for our consequential, domain-heavy forms they all push guidance into the description and keep the placeholder to a bare example.

1

Permanence decides ownership

The description persists; the placeholder evaporates on focus. So the rule writes itself: anything the user must still see while typing, or when they return to verify a value, belongs in the description. The placeholder can only hold what is safe to lose.

2

No hint slot, so the description owns format

We have no hint line below the input — so the description owns format. That leaves it the only persistent home for "10-digit NPI" or "reporting year, YYYY"; a format dropped into the vanishing placeholder is worse than none at all.

3

A placeholder is an example, not an instruction

A placeholder is greyed, ephemeral, and routinely read as a pre-filled value. The only thing it can safely carry is a sample of the answer's shape — and even then, only when that shape is hard to describe in words alone.

4

Restraint keeps the signal

Most fields need neither slot. Padding an obvious field with a description, or echoing its label in a placeholder, is noise — it trains users to stop reading the text on the fields that genuinely need it.

04 / When to leave them out

The default for both is nothing

A description earns its place by saying something the label can't. A placeholder earns its place by showing a shape that words would labour to describe. When neither is true, ship the field with its label alone — an empty description and an empty placeholder are the right answer far more often than designers reach for them.

Leave them out when

  • The label already determines the answer beyond doubt — a person's first name, an email, a plainly named date. No description.
  • The format is obvious or the field is forgiving. Don't manufacture a constraint to fill the description.
  • An example would only repeat the label, or there is no single representative value to show. No placeholder.
  • The value's shape is self-evident from the label and width. The placeholder adds clutter, not clarity.

Reject in review

  • Essential information — a format, an instruction, a constraint — living only in the placeholder, where it vanishes on input.
  • A placeholder that echoes its label, or one so complete it reads as a value the user has already entered.
  • A description that merely restates the label instead of adding meaning, the why, or a format.
  • Format guidance deferred to a placeholder because "there's nowhere else" — there is: the description. We have no hint line, so the description is where it goes.
05 / What the standards say

Persistent helper text, placeholder demoted

The accessibility standard and the major design systems converge: guidance belongs in persistent helper text, and the placeholder is at best an optional example, never the carrier of meaning. The systems differ only on where the persistent text sits — and ours sits above the input, in the description. Links are in the references.

GOV.UK Design System — closest precedent

Don't use placeholders; use persistent hint text

GOV.UK is blunt: do not use placeholder text in place of a label or hint, because it disappears on input, is mistaken for a pre-filled value, has poor contrast, and isn't consistently announced. Format and examples go in hint text — which in their layout sits with the label, above the input. That is precisely our description slot.

guidance above the input
Nielsen Norman Group

Placeholders in form fields are harmful

NN/g's testing shows placeholder-as-guidance increases errors and cognitive load: users forget what was asked, can't verify their answer, and skip fields that look filled. Keep any guidance persistent, and show examples outside the field rather than inside it.

keep guidance persistent
W3C · WCAG — the standard

Instructions must be available, not ephemeral

Success Criterion 3.3.2 (Labels or Instructions) requires that instructions are provided when content needs input. Guidance that exists only in a placeholder fails it — it is not a reliable accessible name, and its low-contrast grey commonly fails 1.4.3 as well.

3.3.2 instructions
IBM · Carbon — closest to our world

Assistive text holds the guidance

Carbon, the system built for data-heavy enterprise products, keeps field guidance in persistent assistive text and treats the placeholder as an optional example only — never the home for essential information. That is the same division we draw, with the persistent text moved to the description above the input.

persistent assistive text
MDN · web standards

Describe via aria-describedby, not the placeholder

MDN documents the mechanics this rule depends on: supplementary description text linked to the input with aria-describedby is announced by assistive tech, whereas the placeholder attribute is not a dependable accessible name. The description is the slot that actually reaches every user.

aria-describedby
Synthesis

The pattern for forms like ours

For a two-slot system with no hint line, the field is unanimous: the description is the persistent helper that carries meaning and format, and the placeholder is a demoted, optional example. VC should follow it, and keep the placeholder empty by default.

06 / Recommendation for VC

Guidance in the description; the placeholder is only an example

The house rule (provisional)

  • Put anything the user must read to fill the field right in the description above the input — meaning, the why, format and constraints — where it persists and is announced via aria-describedby.
  • Use the placeholder only as a faint, skippable example of a well-formed value, and only when an example removes real doubt about the value's shape. Truncate it so it can't be read as a finished entry.
  • Default both off. A self-evident field ships with its label alone; an empty description and placeholder is usually correct.
  • Never put the field's name, an instruction, or essential information in the placeholder. It disappears on input and isn't reliably announced.
  • We have no hint line below the input, so never defer format guidance to a vanishing placeholder — the description is its only home.
