How VC handles
screen width

VC is a desktop application, so its breakpoints live in the desktop band — not a phone-to-desktop ladder. One rule does most of the work: width follows content type. A datagrid uses the width it is given; a form caps to a reading measure.

Our context

VC runs on large desktop monitors — analysts and admins building cohorts, measures and risk models, many screens dense with datagrids and charts. We serve no phone or tablet-portrait viewport, and the layout never exceeds a rail beside a canvas. What decides the breakpoints is not device width but content type, on one 12-column grid, across the desktop band only.

The rule (provisional)

VC is a desktop application. Hold one 12-column grid across the desktop band and define the tiers in rem.

  • Width follows content type — a reading or configuration canvas caps its measure; a datagrid or chart canvas uses the full width.
  • The grid count stays 12 at every tier; column width and the gutter step change, not the count.
  • Narrow widths are met by WCAG zoom reflow, not a designed mobile layout.

The deciding question for any screen: is the canvas read, or is it data? Read caps; data spreads.

03 / Large width on a data canvas

Width buys more data, not bigger charts

When a data canvas gets a wider window, add tiles and columns — do not inflate the charts already on screen. Chart tiles hold a readable minimum width, and the grid packs more of them per row as the tier climbs. In CSS this is one rule, not one per tier: a repeat(auto-fill, minmax(22rem, 1fr)) grid adds columns on its own as the canvas grows.

04 / The reasoning

Six forces behind the rule

The rule is structural, not stylistic. Six forces shape the ladder, and for VC's screens they all point one way.

1

Desktop-first, not mobile-first

We design the desktop band, because that is the only band we serve. Inverting a mobile-first ladder onto a tool no one opens on a phone adds rungs we never use and a base layout we never ship.

2

Width follows content type

The deciding question per screen is whether the canvas is read or is data. A form, a wizard, a settings page caps to a measure; a datagrid, a chart, a dashboard uses the width. One rule settles most layout arguments before they start.

3

One 12-column grid across the band

The count holds at 12; only column width and the gutter step flex per tier. This is the contract a designer and a developer share — the same grid in Figma and in CSS, so a layout reads identically on both sides of the handoff.

4

Reflow is accessibility, not mobile

Narrow widths exist to satisfy zoom, not to serve phones. WCAG 1.4.10 requires content to reflow without a second scroll axis down to 320px effective — which a desktop user hits at 400% zoom. We meet it; we do not design a layout for it.

5

rem, not px

Breakpoints in rem respond to a user's font-size, not just to zoom. An admin who sets a larger default text size shifts the tiers with it; pixel breakpoints would ignore that preference and hold the layout fixed against them.

6

Container queries for components

A reusable component adapts to its slot, not the viewport. A cohort card or a chart that can sit in the rail or fill the canvas should respond to the width it is in — a container query — so the same component is correct in both places.

05 / Edge cases and anti-patterns

Where to draw the line in review

Cap the measure when the canvas is read

  • Forms, wizards, settings, and any prose or single-record detail — cap to the measure token, no matter how wide the window.
  • A single working column in a builder; the section rail carries wayfinding, not a second column of fields.
  • Empty space to the right of a capped canvas is correct, not a layout bug to fill.

Use the width when the canvas is data

  • Datagrids, dashboards, and chart canvases spread to all 12 columns and add tiles, columns or detail as the window grows.
  • Chart tiles hold a readable minimum width and pack more per row (2-up → 3-up → 4-up); they do not stretch to fill.
  • A single chart caps at a sensible maximum, then spends extra width on more time range or series — never wider bars.
  • Density steps up from Standard to Large to Ultrawide; the layout does not reshape, it just shows more.
  • Reusable data components respond to their container, so a grid is right whether it is in the rail or the full canvas.

Reject in review

  • A phone or tablet-portrait layout for a viewport VC does not serve. The narrow case is reflow for zoom, and that is automatic, not designed.
  • A "mobile" breakpoint below the desktop floor, or a hamburger nav standing in for the rail. We have no mobile rung.
  • Device-named breakpoints — "mobile", "tablet", "desktop". Name the tier by its place in the band, not by a device.
  • Pixel breakpoints. They ignore a user's font-size; define the tiers in rem.
  • A reading canvas stretched to the full window so the line length runs past the measure. Cap it.
  • A single chart stretched across the full window until the trend flattens. Cap the chart and fill the row with companion tiles or more data.
  • A new breakpoint added to fix one screen. Reach for a container query on that component first; a new global tier is a standards change, not a local patch.
06 / What the major design systems say

The enterprise convention, adapted to a desktop tool

The systems agree on the mechanics and diverge on the band. All define a small set of min-width tiers; the enterprise, data-dense ones run a 12-column grid much like ours, while the consumer-facing ones carry mobile rungs we drop. Carbon is the closest in spirit; Bootstrap is the closest on the grid. Links are in the references.

IBM · Carbon — closest in spirit

A data-dense grid with a constant gutter

Carbon's 2x Grid is built for the same data-heavy enterprise work as ours: explicit sm/md/lg/xlg/max breakpoints, a column count that steps up with width, and a 32px gutter held constant at every breakpoint so columns stay predictable. We take the stance and the constant-gutter discipline; we run 12 columns rather than Carbon's 16, because that is VC's Figma grid.

data-dense · constant gutter · 16-col
Bootstrap 5 — closest on the grid

A 12-column grid on min-width tiers

Bootstrap's grid is 12 columns across six min-width breakpoints from 576 to 1400px, each with a container max-width that caps content rather than letting it stretch. The 12-column structure and the capped container are exactly VC's model; we keep only the upper, desktop tiers and express them in rem.

12-col · min-width · capped container
Google · Material 3

Window size classes, not device names

Material replaced device breakpoints with window size classes — Compact, Medium, Expanded, Large, Extra-large — naming the layout intent rather than a phone or tablet. That is the naming discipline we adopt: a tier is a place in the band, not a device.

semantic classes over device names
Tailwind CSS v4

rem breakpoints, by default

Tailwind defines its sm–2xl breakpoints in rem — 40, 48, 64, 80, 96rem — so they track a user's root font-size, and exposes them as CSS variables. We do not use the framework, but we follow the unit: rem tiers, for the accessibility reason, are the current default for good cause.

rem tiers for accessibility
GitHub · Primer

Two layers: ranges for design, breakpoints for build

Primer separates viewport ranges — narrow, regular, wide — that a designer reasons about from the finer xsmall–xxlarge breakpoints a developer writes against. That two-layer split is precisely the designer-and-developer handoff this record is built to serve.

ranges for design · breakpoints for build
Synthesis

The pattern for a tool like ours

For a desktop-only, datagrid- and chart-heavy tool on a 12-column grid, the field points one way: a short ladder of rem tiers in the desktop band, a constant-gutter 12-column grid throughout, and width decided by content type. Our usage should follow it.

07 / Recommendation for VC

One grid, content-led width, desktop band only

The house rule

  • Width follows content type. A reading or configuration canvas — forms, wizards, settings, single-record detail — caps to the measure token. A datagrid, dashboard or chart canvas uses the full width and densifies. This is the deciding question for every screen.
  • One 12-column grid across the whole desktop band. The count stays 12 at every tier; column width and the gutter step flex, the structure does not. It is the same grid in Figma and in CSS.
  • On a data canvas, add columns, not bigger charts. Chart tiles hold a readable minimum and pack more per row as width grows, 2-up to 4-up; a single chart caps to a sensible maximum and spends the extra width on more time range or series, never wider bars.
  • Define the tiers in rem — Base 64rem, Standard 80rem, Large 90rem, Ultrawide 100rem — so they track a user's font-size, not only zoom.
  • Author at Standard (1280px) as the base desktop target. Large and Ultrawide add density, not new layouts; Base collapses the rail to an icon strip.
  • Container queries carry reusable components. A component that can sit in the rail or fill the canvas responds to its container, not the viewport.
  • Mobile and tablet-portrait are out of scope, met only by WCAG zoom reflow down to 320px effective. Do not draw a phone layout, add a mobile breakpoint, or name a tier after a device.
