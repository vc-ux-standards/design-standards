Bars: orientation,
order, and too many

The bar chart is VC's default for comparison and ranking, so most of the questions are about discipline: which way it runs, how it's ordered, and what to do when the categories overflow. Sort by value, start at zero, go horizontal when labels are long — and past ~12 bars, show the top and group the rest.

Our context

VC ranks things all day — measures by performance, providers by gap count, contracts by spend — and the category count is rarely tidy. A registry can return dozens of measures. The deciding pressure is legibility under real cardinality: a chart that reads at 6 bars must not silently become unreadable at 40.

The rule

Use a column (vertical) for a short, ordered series like a monthly trend; a horizontal bar for ranking and for long category labels. Sort by value unless a natural order is the point, and start the axis at zero — always, for bars. Past ~12 categories, show the top-N and roll the tail into a labelled "Other"; the full set belongs in a sortable table.

01 / Quick reference

Column or horizontal bar

Orientation is a reading decision, not a style one. Columns are for an ordered axis you read left to right; horizontal bars are for ranking and for labels that need room. When in doubt in VC, it's a horizontal bar.

Column (vertical) Horizontal bar
Category count Up to ~8 that fit without crowding. Many — rows stack downward and scroll.
Labels Short, or they rotate and truncate. Long domain names sit left, on one line.
Reading task Change across an ordered axis, usually time. Ranking and magnitude comparison.
Order Keep the natural order — the axis is the point. Sort by value, biggest first.
Best fit in our app Readmissions per month; enrolment by quarter. Measures, providers or contracts ranked.
02 / The pattern in practice

Overflow, and the zero baseline

Two failures account for most bad bars in review: too many of them, and an axis that doesn't start at zero. Both are visible the moment the fix sits beside them.

Avoid
Quality measures · performance

Twenty-eight measures crammed into one column block, unsorted. Bars this thin can't be labelled or ranked; the reader can't find the worst performer or read a single value. Width, not the question, is setting the count.

Use
Quality measures · top performers

Sorted, top eight, and the tail rolled into a greyed "Other". Each domain label gets the full measure, values sit at the bar end, and the ranking reads top to bottom. Past ~12 categories, this is the move — and the full list stays a click away as a sortable table.

Avoid
Screening rate · axis from 90%

An axis starting at 90% turns a three-point spread into a cliff. Four screening rates within three points of each other look wildly unequal — bar length no longer matches value, which is the one job a bar has.

Use
Screening rate · axis from 0%

Zero baseline: the bars tell the truth. The same four rates are nearly level, because bar length is only honest from zero. If a three-point difference matters, say it in a number or a dot plot — don't fake it with a cropped axis.

03 / The reasoning

Four rules the bar earns

A bar encodes one value as one length. Every rule below protects that: keep the length honest, order it so the eye can read it, and don't ask it to carry more categories than it can show.

1

Zero baseline, always

A bar's length is its value. Crop the axis and length stops matching value, which reads as a lie. A non-zero axis is a line-chart liberty; a bar never takes it.

2

Sort by value, not the alphabet

Ranking is the whole point of most bar charts. Order biggest-to-smallest so the rank reads directly. Keep a natural order only when that order — time, a sequence — is itself the question.

3

Horizontal when labels or counts grow

Columns divide the width by the category count and truncate the labels. Horizontal bars stack downward, scroll, and give each domain name the full measure — which is why VC's rankings are almost always horizontal.

4

Cap the count, group the tail

Past ~12 categories a bar chart stops comparing. Show the top-N, roll the rest into a labelled "Other", and put the exhaustive list in a sortable table. Never squeeze 40 bars into a tile.

04 / What the field says

The bar conventions are settled

Sort by value, start at zero, and don't overload — every major source agrees. The disagreements in dataviz are elsewhere; the bar chart is the settled ground these standards stand on.

IBM · Carbon — closest to our world

Bar defaults, sorted, zero-based

Carbon defaults categorical comparison to bar charts, recommends ordering by value for ranking, and uses horizontal bars when category labels are long — the exact shape of VC's measure and provider rankings. Its axes start at zero.

sorted, horizontal, zero-based
Datawrapper Academy

Order by value; group small categories

Datawrapper's bar guidance is explicit: sort bars by value unless the order carries meaning, always start the axis at zero, and combine small categories into an "Other" group rather than showing a long unreadable tail.

sort · zero · group the tail
FT · Visual Vocabulary

Ranking is its own family

The FT treats "ranking" as a distinct purpose served by ordered bars, columns and lollipops — bars sorted by value, the largest read first. It is where the horizontal-bar-for-ranking default comes from.

ordered bars for ranking
Claus Wilke · Fundamentals

Bars must start at zero; dot plots for many

Wilke is categorical that bar length demands a zero baseline, and recommends dot plots when a category list grows long or the values cluster in a narrow band — the honest alternative to a cropped axis.

zero baseline; dot plot at scale
Google · Material

Bar for category comparison

Material recommends bars for comparing values across categories and, like the others, anchors the value axis at zero so the length reads as the quantity.

category comparison, zero axis
Synthesis

The pattern for tools like ours

For VC's ranked measures, providers and contracts at real cardinality: horizontal bars, sorted by value, on a zero baseline, capped with a grouped "Other" and backed by a sortable table. Our usage should follow it.

the VC bar default
