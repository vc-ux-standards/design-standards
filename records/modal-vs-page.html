Modal, full page,
or side panel

A first pass, pending the precedent and sourcing work the stepper record already has. The rule and the table below are provisional, meant to be argued with, not cited yet.

Our context

The same as everywhere in Value Connect: desktop, analyst and admin users, long server-backed configuration with content that lives in a canvas. That sends most heavy work to full pages, leaves modals for short interruptions, and makes the side panel the tool for work that has to sit beside the main view.

The rule (provisional)

Use a modal only for a short, self-contained task that must interrupt: a confirmation, a quick create, a decision the user has to resolve before continuing. Use a full page for anything long, multi-section, or that the user navigates within and returns to. Use a side panel when the task supports the main view and that context needs to stay visible. The deciding question is whether the task needs brief and total attention, is a workspace in its own right, or runs alongside the main work.

02 / Reasoning (draft)

Why the task decides, not the visual weight

The common error is reaching for a modal because it feels lightweight, then loading it with a multi-step form that needs scrolling, validation, and the context the modal has hidden behind it. A modal removes the page behind it, so anything that needs that context, or runs long enough that the user loses their place, belongs on a full page or in a side panel. The reverse error is sending a one-line confirmation to a full page and throwing away the user's place in their work. Match the container to the shape of the task: brief and blocking, a workspace, or a companion to the main view.
