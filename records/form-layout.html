Draft Standards · Forms

Form layout on
wide artboards

A 1400px artboard is a canvas, not a form width. This record sets where a form sits on the page and how wide each field should be, so a form stops sprawling across the whole screen and a date stops stretching a thousand pixels edge to edge. Draft: argue with the numbers, don't cite them yet.

Our context

With the whole width of a 1400px artboard to fill, a form tends to sprawl the full page — one full-width input per row, whether the field holds a ZIP code or a paragraph. The fix isn't responsive trickery: it's a capped form column with each field sized to its content.

The rule (provisional)

Size the layout to the task, not the artboard. A sequential form — setup, onboarding — sits in a single column capped at 640px, centred. A dense record edited all day earns a grouped grid sized to its content. Either way, size every input to the content it holds — a date, code or amount is never full width.

01 / Page placement

Place the form by its task, not the artboard

The first decision is where the form lives on the page, before a single field. The failure to avoid is the same at every scale: a form stretched to fill the page width, the way a field stretched to fill its row reads as a wall — lines run past 90 characters and the eye loses the left edge on the return sweep. What replaces it depends on the task. A sequential form belongs in a single column capped at 640px (40rem), held there however wide the screen gets — the case the comparison below shows. A dense record an analyst edits daily belongs in a structured, grouped grid sized to its content (see §04). Neither one fills the artboard.

Avoid
100% — full page width

The form fills the page. Labels and inputs run the full width, lines get too long to track, and there is no stable column to scan down.

Use
640px column

A 640px column, centred in the content area, with the page width as breathing room. The form reads as one calm path regardless of how wide the screen gets.

Where the column sits: centred in the content area

Centre the column in the content area — the space that remains after any chrome. One rule for every screen: the column keeps balanced margins on both sides, so the form reads as the deliberate centre of attention rather than something shoved against an edge. The two extremes the hero shows are what it must not do: stretch to fill the width, or sit jammed against the chrome with all the empty space dumped on one side.

With page chrome: sidebar and steppers

Our pages carry chrome — the ~200px navigation sidebar on every page, and a stepper on the longer flows. The rule doesn't change: the 640px column sits centred in the content area — the space that remains beside or below whatever chrome precedes it. The stepper's own orientation is a separate decision; see Stepper orientation.

02 / Quick reference

Maximum input width by field type

Inside that column, width signals expected length: a field sized to its content tells the user how much to type before they read the label. Cap each input near the longest realistic value, not the column. Widths below are maximums in ch (character widths) with an approximate pixel size at our field font.

Field typeMax widthWhy that size
Day / short integer≈ 5ch (~45px)One or two digits. A wider box invites a wrong value.
Code / PIN / ZIP (4–6 digit)≈ 7ch (~60px)Fixed-length token; the box should look fixed-length too.
Date (MM/DD/YYYY)≈ 12ch (~110px)Ten characters plus separators. Never edge to edge.
Currency / amount≈ 12ch (~110px)A few digits and a decimal; align these right in tables.
Phone≈ 16ch (~145px)Holds a formatted number with country code.
Name (first / last)≈ 24ch (~215px)Comfortable for most single names; two of these pair into a row.
Email≈ 30ch (~270px)Most addresses fit; the rest scroll within the field.

Sizes are maximums, not targets — a field may be narrower. The principle, not the exact pixel, is the rule: a fixed-length value gets a fixed-looking box, free text gets a readable line, and nothing is sized by the artboard.

03 / Field width

Size the box to the value

The headline failure: every input stretched the full width, so a five-digit code and a date each get a box wide enough for a sentence. It looks tidy in a mockup and reads as noise in use. Match the box to the longest realistic value instead.

04 / One column versus two

Columns for structure, not to fill space

A single column gives one unambiguous path down the form: in a sequential flow it is faster to complete and harder to misread. The failure is the arbitrary second column — unrelated fields paired across a gutter to use up the artboard, so the eye zig-zags, skips fields, and reads relationships that don't exist. In a sequential form, use a second column only for fields that are genuinely paired and short — first and last name, city and state, start and end date — and never split one logical field across two columns.

Density is the exception that earns the grid. A record an analyst opens dozens of times a day — a provider or contract with twenty-plus fields — is slower to scan in one tall column than in a structured grid: deliberate column groups, related fields together, each still sized to its content. That is not the artboard-filling two-column this section warns against; it is the dense-record regime from §01, and it is what systems built for data-heavy work (§08) actually do. The test is whether the second column carries structure or merely fills space.

05 / Grouping, labels and spacing rhythm

Make structure visible

A long form needs visible structure: related fields gathered under a heading, labels sitting directly above their inputs, and a spacing rhythm that ties each label to its field while separating one group from the next. Top-aligned labels scan fastest and survive long or translated text; a flat, even spacing with no grouping turns the form into an undifferentiated list.

The rhythm, in three steps: label → input ≈ 6px (they belong together), field → field ≈ 22px, group → group ≈ 44px (double the inter-field gap, plus a heading).

06 / Choice controls

Radios and checkboxes: stack them

Choice controls follow the same single-column logic as fields. Stack options vertically so they share a left edge and read as one set; reserve horizontal and multi-column layouts for the few cases that earn them. The number of options decides the shape.

Four radio buttons → one per line

A small, mutually exclusive set wants every option visible at once. Stack the radios vertically; don't lay them in a row, and don't bury a short, important choice inside a dropdown.

07 / Keeping long forms scannable

One concept per step; each step laid out to its task

The placement and width rules carry into long, multi-step forms — applied per step, not flattened across all of them. Break the work into steps of one concept each, lay out each step to its own task (a sequential step in a capped column, a dense record in a structured grid, per §01 and §04), and let the stepper — not a wall of fields — carry the sense of progress. A multi-step flow is not automatically sequential: a step can be a dense record. When a step itself runs long, the named groups from §05 are the seams to scan by.

For a long or multi-step form

  • Lay out each step to its task — a sequential step in a single ~640px column, a dense record in a structured grid. Don't force one shape on every step; the canvas around either can be wide.
  • One concept per step rather than one giant scroll.
  • Group fields within a step under headings; the group is the unit a user scans for.
  • Keep field widths content-sized inside steps too; consistency across steps is what makes the whole flow feel calm.

Reject in review

  • A form that fills the page, or a sequential-form column that grows with the viewport instead of staying capped at 640px.
  • A form column jammed against the chrome or stretched full width, rather than centred in the content area with balanced margins.
  • Any input sized by the artboard rather than its content — the full-width date, code or amount.
  • A second column used to fill width, pairing fields that aren't related.
  • Radios or checkboxes laid in a row, or a long checkbox set left as one ungrouped scroll.
08 / What the major design systems say

A capped single column is the convention

This is not a house preference. The systems and research that production forms are built on converge on the moves that matter: never let the form fill the artboard, size inputs to their content, and choose column count by the task — a single column for sequential flows, a structured grid for dense records. Links are in the references.

GOV.UK Design System — closest precedent

Match the input to the content

GOV.UK gives every text input a set of width-override classes — both fixed character widths (2, 3, 4, 5, 10, 20 characters) and fluid fractions (full, three-quarters, one-half, one-third, one-quarter) — and the guidance is explicit: make the size of an input proportionate to the information it requires. A field for a few digits should be visibly small.

size to content
Baymard Institute — checkout research

Single column completes faster

Baymard's large-scale form usability testing finds multi-column layouts cause users to misinterpret field relationships and skip fields, and recommends a single-column flow for the main path, reserving side-by-side fields for short, clearly related pairs. A single column gives one reading path.

single column
Typographic measure

Line length caps the column

Readable line length tops out around 50–75 characters; past that the eye struggles to find the start of the next line. A form column much wider than ~640px pushes labels and help text beyond that measure, which is the ceiling behind the 640px cap — the same logic that bounds body text bounds a form.

~640px column
Nielsen Norman Group · Luke Wroblewski

Labels above; small sets as radios

Eye-tracking work popularised by Wroblewski's Web Form Design and echoed by NN/g shows top-aligned labels are fastest to scan, and that a small set of mutually exclusive options belongs in a vertical list of radios — visible at once — rather than a dropdown that hides them.

top labels, stacked radios
IBM · Carbon — closest to our world

Consistent spacing and field sizing

Carbon, the system for data-heavy enterprise products, specifies form field widths and a consistent vertical spacing scale, and lays dense records out in structured multi-column grids rather than one tall column. It is the nearest match to VC's density — the authority for the dense-record regime — and it confirms the rhythm in §05.

fixed widths, spacing scale
Synthesis

The pattern for forms like ours

The systems split cleanly by task. For a sequential flow the consumer-form research holds: a single ~640px column, centred, choices stacked. For the dense records an analyst edits daily, Carbon's stance holds: a structured, grouped grid sized to content. Both agree on what matters everywhere — inputs sized to their content, labels above, fields in named groups — and both reject the form that fills the artboard.

task-fit width, content-sized fields
09 / Recommendation

Size the layout to the task; size every field to its content

One rule with two branches, not two rules. The decision is a single question asked once per screen — is this a sequential flow or a dense record? A uniform "640px, always" is easier to police in review, but it makes the analyst's daily record-editing slower, and that trade isn't worth it. Branch on the task once, then everything below applies to both.

The house rule (provisional)

  • Never let the form fill the artboard. The canvas is workspace; the form is sized to its task. This is the one non-negotiable.
  • Sequential flows → a single column capped at 640px (40rem), centred in the content area — beside the sidebar, below or beside a stepper. Never jammed against the chrome.
  • Dense records → a structured, grouped grid sized to its content groups, the Carbon-aligned case. Columns must carry structure, not fill space; never split one logical field, never pair unrelated fields to use up width.
  • Size every input to its content in either layout, using the §02 table as the ceiling. A code, day, date or amount is never full width.
  • Labels above inputs, gathered into named groups, with the spacing rhythm of §05 (≈6 / 22 / 44px).
  • Stack choice controls. Radios and short checkbox sets one per line; past ~7 checkboxes, split into two columns or group under sub-headings.
  • In long flows, one concept per step, content-sized fields inside each step; let the stepper carry progress.
