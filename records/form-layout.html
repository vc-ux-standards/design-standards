Draft Standards · Forms

Form layout on
wide artboards

A 1400px artboard is a canvas, not a form width. This record sets where a form sits on the page and how wide each field should be, so a form stops sprawling across the whole screen and a date stops stretching a thousand pixels edge to edge. Draft: argue with the numbers, don't cite them yet.

Our context

Designers lay out on 1400px artboards and, with the whole width to fill, let the form sprawl the full page and default to one input per row stretched the full measure — a 90s-web shape. The result reads as a wall of identical full-width boxes regardless of whether the field holds a ZIP code or a paragraph. VC is a desktop, analyst- and admin-facing configuration tool with a persistent navigation sidebar of about 200px on every page, and steppers on the longer flows, so the question of where the form column sits relative to that chrome comes up constantly. The fix is not responsive trickery — it is placing the form in a capped column and sizing each field to its content.

The rule (provisional)

Put every form in a single column capped at 640px (40rem), centred in the content area — never stretched across the page. The canvas around it can be any width; the column is not. Then size every input to the content it holds, not the column: a date, a code, an amount or a day is never full width. One column by default, a second only for genuinely paired fields; labels above their inputs; a consistent spacing rhythm so a long form stays scannable.

01 / Page placement

Put the form in a 640px column

The first decision is where the form lives on the page, before a single field. A form that fills the page width is the same mistake as a field that fills its row, one level up: lines run past 90 characters, the eye loses the left edge on the return sweep, and the page reads as a wall. Put every form in a single column capped at 640px (40rem) and hold it there. Below that, a shorter form can be narrower; never wider.

Avoid
100% — full page width

The form fills the page. Labels and inputs run the full width, lines get too long to track, and there is no stable column to scan down.

Use
640px column

A 640px column, centred in the content area, with the page width as breathing room. The form reads as one calm path regardless of how wide the screen gets.

Where the column sits: centred in the content area

Centre the column in the content area — the space that remains after any chrome. One rule for every screen: the column keeps balanced margins on both sides, so the form reads as the deliberate centre of attention rather than something shoved against an edge. The two extremes the hero shows are what it must not do: stretch to fill the width, or sit jammed against the chrome with all the empty space dumped on one side.

With page chrome: sidebar and steppers

Our pages carry chrome — the ~200px navigation sidebar on every page, and a stepper on the longer flows. The rule doesn't change: the 640px column sits centred in the content area — the space that remains beside or below whatever chrome precedes it. The stepper's own orientation is a separate decision; see Stepper orientation.

02 / Quick reference

Maximum input width by field type

Inside that column, width signals expected length: a field sized to its content tells the user how much to type before they read the label. Cap each input near the longest realistic value, not the column. Widths below are maximums in ch (character widths) with an approximate pixel size at our field font.

Date (DD/MM/YYYY)≈ 12ch (~110px)Ten characters plus separators. Never edge to edge.
Currency / amount≈ 12ch (~110px)A few digits and a decimal; align these right in tables.

Sizes are maximums, not targets — a field may be narrower. The principle, not the exact pixel, is the rule: a fixed-length value gets a fixed-looking box, free text gets a readable line, and nothing is sized by the artboard.

03 / Field width

Size the box to the value

The headline failure: every input stretched the full width, so a five-digit code and a date each get a box wide enough for a sentence. It looks tidy in a mockup and reads as noise in use. Match the box to the longest realistic value instead.

04 / One column versus two

Single column, with paired fields the only exception

A single column gives one unambiguous path down the form, which is faster to complete and harder to misread. Two columns exist to fill the artboard, and they break the reading path: the eye zig-zags and skips fields. Use a second column only for fields that are genuinely paired and short — first and last name, city and state, start and end date — and never split one logical field across two columns.

05 / Grouping, labels and spacing rhythm

Make structure visible

A long form needs visible structure: related fields gathered under a heading, labels sitting directly above their inputs, and a spacing rhythm that ties each label to its field while separating one group from the next. Top-aligned labels scan fastest and survive long or translated text; a flat, even spacing with no grouping turns the form into an undifferentiated list.

The rhythm, in three steps: label → input ≈ 6px (they belong together), field → field ≈ 22px, group → group ≈ 44px (double the inter-field gap, plus a heading).

06 / Choice controls

Radios and checkboxes: stack them

Choice controls follow the same single-column logic as fields. Stack options vertically so they share a left edge and read as one set; reserve horizontal and multi-column layouts for the few cases that earn them. The number of options decides the shape.

Four radio buttons → one per line

A small, mutually exclusive set wants every option visible at once. Stack the radios vertically; don't lay them in a row, and don't bury a short, important choice inside a dropdown.

07 / Keeping long forms scannable

One concept per step, the column held narrow

The placement and width rules carry into long, multi-step forms. Break the work into steps of one concept each, keep the same 640px column inside every step, and let the stepper — not a wall of fields — carry the sense of progress. When a step itself runs long, the named groups from §05 are the seams to scan by.

For a long or multi-step form

  • Hold the 640px column inside every step; the canvas around it can be wide, the column is not.
  • One concept per step — details, then population, then logic — rather than one giant scroll.
  • Group fields within a step under headings; the group is the unit a user scans for.
  • Keep field widths content-sized inside steps too; consistency across steps is what makes the whole flow feel calm.

Reject in review

  • A form that fills the page, or a column that grows with the viewport instead of staying capped at 640px.
  • A form column jammed against the chrome or stretched full width, rather than centred in the content area with balanced margins.
  • Any input sized by the artboard rather than its content — the full-width date, code or amount.
  • A second column used to fill width, pairing fields that aren't related.
  • Radios or checkboxes laid in a row, or a long checkbox set left as one ungrouped scroll.
08 / What the major design systems say

A capped single column is the convention

This is not a house preference. The systems and research that production forms are built on converge on the same moves: hold the form to a readable column, prefer a single column, and size inputs to their content. Links are in the references.

GOV.UK Design System — closest precedent

Match the input to the content

GOV.UK gives every text input a set of width-override classes — both fixed character widths (2, 3, 4, 5, 10, 20 characters) and fluid fractions (full, three-quarters, one-half, one-third, one-quarter) — and the guidance is explicit: make the size of an input proportionate to the information it requires. A field for a few digits should be visibly small.

size to content
Baymard Institute — checkout research

Single column completes faster

Baymard's large-scale form usability testing finds multi-column layouts cause users to misinterpret field relationships and skip fields, and recommends a single-column flow for the main path, reserving side-by-side fields for short, clearly related pairs. A single column gives one reading path.

single column
Typographic measure

Line length caps the column

Readable line length tops out around 50–75 characters; past that the eye struggles to find the start of the next line. A form column much wider than ~640px pushes labels and help text beyond that measure, which is the ceiling behind the 640px cap — the same logic that bounds body text bounds a form.

~640px column
Nielsen Norman Group · Luke Wroblewski

Labels above; small sets as radios

Eye-tracking work popularised by Wroblewski's Web Form Design and echoed by NN/g shows top-aligned labels are fastest to scan, and that a small set of mutually exclusive options belongs in a vertical list of radios — visible at once — rather than a dropdown that hides them.

top labels, stacked radios
IBM · Carbon — closest to our world

Consistent spacing and field sizing

Carbon, the system for data-heavy enterprise products, specifies form field widths and a consistent vertical spacing scale between fields and groups rather than stretching inputs to the container. It is the nearest match to VC's density and confirms the rhythm in §05.

fixed widths, spacing scale
Synthesis

The pattern for forms like ours

For a desktop, analyst-facing configuration tool, the field points one way: a single 640px column, centred in the content area, inputs sized to their content, labels above, fields gathered into named groups, choices stacked. The artboard width is workspace, not form width.

09 / Recommendation for VC

One 640px column, centred; size to content

The house rule (provisional)

  • Put the form in a single column capped at 640px (40rem), centred in the content area — the space beside the sidebar, below or beside a stepper. Never full-page, never jammed against the chrome.
  • Size every input to its content, using the §02 table as the ceiling. A code, day, date or amount is never full width.
  • One column by default. A second column only for short, genuinely paired fields; never split one logical field, never two-column to fill space.
  • Labels above inputs, gathered into named groups, with the spacing rhythm of §05 (≈6 / 22 / 44px).
  • Stack choice controls. Radios and short checkbox sets one per line; past ~7 checkboxes, split into two columns or group under sub-headings.
  • In long flows, one concept per step, the same 640px column and content-sized fields inside each step; let the stepper carry progress.
